# Supply Chain

PQC Bridge should become a supply-chain-conscious cryptography project before
stable release.

## Current State

v0.1 includes:

- lockfiles for Rust and Node.js
- Dependabot configuration
- cargo-deny configuration
- GitHub Actions CI
- release workflow scaffold

## Required Before Stable Release

- SBOM generation
- release checksums
- signed git tags
- signed release artifacts or artifact attestations
- provenance attestations where practical
- dependency review workflow
- documented package publishing ownership
- reproducible build notes for native artifacts

## Release Integrity Baseline

The release workflow blocks source material upload until the Rust, binding,
package dry-run, C/C++ header, Swift, SBOM, and checksum gates pass.

Current release source materials:

- `pqc-bridge-source.tar.gz`
- `SHA256SUMS`
- `sbom.spdx.json`

`SHA256SUMS` records SHA-256 digests for the source archive and SBOM. Release
notes must include the checksum file contents or link to the uploaded artifact.

Git tags for stable releases must be signed by an authorized maintainer key.
Unsigned release tags are not acceptable for `v1.0.0` or later. The release
operator must verify the tag locally before publishing release notes:

```sh
git fetch --tags origin
git tag -v vX.Y.Z
```

GitHub artifact attestations are generated by the release workflow using the
repository OIDC identity and Sigstore-backed signing. The workflow creates:

- a provenance attestation for subjects listed in `SHA256SUMS`
- an SPDX SBOM attestation for `pqc-bridge-source.tar.gz`

Online verification for released source materials:

```sh
shasum -a 256 -c SHA256SUMS
gh attestation verify pqc-bridge-source.tar.gz -R VrianCao/pqc-bridge
gh attestation verify pqc-bridge-source.tar.gz \
  -R VrianCao/pqc-bridge \
  --predicate-type https://spdx.dev/Document/v2.3
```

Package-registry provenance features should be enabled before publishing
registry packages when the registry supports them. Until registry publishing is
implemented, package dry runs are verification gates and source archive
attestations are the release provenance baseline.

Detached signatures for native artifacts are not yet produced because native
binary artifacts are not published. When native release artifacts are added,
the release task must add detached signatures or Sigstore bundle artifacts for
each native archive before the artifact is considered stable-release ready.

## Package Registries

Planned packages:

| Registry | Package material | Owner requirement | Release status |
| --- | --- | --- | --- |
| crates.io | Rust crates and CLI | Maintainers group with at least two owners and 2FA. | Dry-run only. |
| npm | Node.js package | `@pqcbridge` organization with at least two owners and 2FA. | Dry-run only. |
| PyPI | Python package | Project owner group with trusted publishing and 2FA. | Dry-run only. |
| Maven Central | Java/Kotlin artifacts | Organization namespace controlled by maintainers. | Planned. |
| Swift Package Manager | GitHub tag-based package | Signed GitHub release tag. | Build check only. |
| GitHub Releases | source and future native C/C++ artifacts | Repository maintainers; release artifacts require checksums and attestations. | Source archive only. |

No package may be published from a maintainer's personal account as the sole
owner. Registry credentials must not be stored in repository secrets until the
publishing workflow exists and uses least-privilege, environment-protected
release jobs.

## Backend Provenance

Every backend integration must document:

- upstream repository
- exact version or commit
- license
- maintainer status
- KAT source
- platform support
- known limitations
